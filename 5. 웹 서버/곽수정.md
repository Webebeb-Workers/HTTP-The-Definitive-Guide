# 5. 웹 서버

웹 서버의 종류와 HTTP 통신 진단 방법 (perl), 웹 서버가 어떻게 HTTP 트랜잭션을 처리하는지 알아보는 챕터

## 5.1 다채로운 웹 서버

- **웹 서버**: 웹 서버 SW, 웹 페에지 제공에 특화된 장비
  - HTTP 요청 받아 콘텐츠를 클라이언트에게 돌려주는 역할

### 5.1.1 웹 서버 구현

- 웹 서버
  1. HTTP 프로토콜 구현
  2. 웹 리소스 관리
  3. 웹 서버 관리 기능 제공
- HTTP 프로토콜을 구현하고, 웹 리소스를 관리하고, 웹 서버 관리 기능을 제공하며 TCP 커넥션 관리에 대한 책임을 운영체제와 나눠 가짐

### 5.1.2 다목적 소프트웨어 웹 서버

- 모든 인터넷 웹 사이트의 37%가 마이크로소프트 웹 서버를 통해 서비스되고 있다
- 아파치 웹 서버가 점유율 35%로, 최근 근소한 차이로 점유율 2위로 밀려났다.
- nginx 서버를 사용하는 사이트가 최근 수년간 꾸준히 증가하여 약 14%를 점유하고 있음

### 5.1.3 임베디드 웹 서버

일반 소비자용 제품에 내장될 목적으로 만들어진 작은 웹 서버

## 5.2 간단한 펄 웹 서버

- HTTP/1.1의 기능을 지원하려면 풍부한 리소스 지원, 가상 호스팅, 접근 제어, 로깅, 설정, 모니터링, 그 외 성능을 위한 각종 기능이 필요함

- 그러나 최소한으로 기능하는 HTTP 서버라면 30줄 이하의 펄Perl 코드로 만들 수 있음 \*Perl(Practical Extraction and Reporting Language)

  - 텍스트 처리 및 시스템 관리 등 다양한 용도로 사용되는 고급 프로그래밍 언어

- type-o-serve
  - 클라이언트와 프락시 간의 상호작용 테스트에 유용한 펄 진단 툴

## 5.3 진짜 웹 서버가 하는 일

1. 커넥션을 맺는다 - 클라이언트의 접속을 받아들이거나, 원치 않는 클라이언트라면 닫는다.
2. 요청을 받는다 - HTTP 요청 메시지를 네트워크로부터 읽어 들인다.
3. 요청을 처리한다 - 요청 메시지를 해석하고 행동을 취한다.
4. 리소스에 접근한다 - 메시지에서 지정한 리소스에 접근한다.
5. 응답을 만든다 - 올바른 헤더를 포함한 HTTP 응답 메시지를 생성한다.
6. 응답을 보낸다 - 응답을 클라이언트에게 돌려준다.
7. 트랜잭션을 로그로 남긴다 - 로그파일에 트랜잭션 완료에 대한 기록을 남긴다.

## 5.4 단계 1: 클라이언트 커넥션 수락

- 클라이언트가 지속 커넥션 갖고 있다면 해당 커넥션을 사용하고, 그렇지 않다면 새 커넥션 열어야함

### 5.4.1 새 커넥션 다루기

1. 클라이언트 ➡️ 웹 서버 TCP 커넥션 요청
2. 웹 서버는 커넥션을 맺고, TCP 커넥션에서 IP 주소 추출 ➡️ 커넥션 맞은편의 클라이언트 확인함
3. 웹 서버는 커넥션 목록에 새 커넥션 추가함, 데이터 지켜보기 위한 준비함

- 웹 서버는 어떤 커넥션이든 거절/수락 가능 (by 신원 식별 기법)

### 5.4.2 클라이언트 호스트 명 식별

1. 웹서버: **역방향 DNS**를 사용해 클라이언트의 IP 주소 ➡️ 호스트 명으로 변환함

- = hostname lookup (시간 소요 상당할 수 있음)
- 구체적인 접근 제어 및 로깅을 위해 사용됨

### 5.4.3 ident를 통해 클라이언트 사용자 알아내기

- **ident 프로토콜**: 서버에게 어떤 사용자 이름이 HTTP 커넥션 초기화 했는지 찾아낼 수 있게 해줌
  - 웹 서버가 접속한 클라이언트의 사용자 정보를 확인하기 위함
  - 주로 로깅(logging)에 활용되어 누가 어떤 요청을 했는지 기록
  - 일반 로그 포맷 (Common Log Format) 두 번째 필드는 각 HTTP 요청의 ident 사용자 이름을 담고 있음

```
[클라이언트]                    [웹 서버]
     |                           |
     |-----(1) HTTP 요청 -------->|
     |      (포트 80)             |
     |                           |
     |<---(2) ident 요청 ---------|
     |      (포트 113)            |
     |                           |
     |-----(3) 사용자 정보 -------->|
     |                           |
     |<---(4) HTTP 응답 ----------|
```

1. 단계 1: 클라이언트가 웹 서버에 HTTP 연결 시도 (예: 포트 80 사용)
2. 단계 2: 웹 서버는 클라이언트의 113번 포트로 ident 요청
3. 단계 3: 클라이언트의 ident 서버가 해당 연결의 사용자 정보 응답
4. 단계 4: 웹 서버는 받은 사용자 정보를 로그에 기록하고 HTTP 요청 처리

<img width="578" alt="image" src="https://github.com/user-attachments/assets/c040ae89-1b9f-4345-ba34-0b21ae0d854a" />

- ident 조직 내부에서는 잘 사용할 수 있지만, 공공 인터넷에서는 잘 동작하지 않음
  - 많은 클라이언트 PC는 identd 신원확인 프로토콜 데몬 소프트웨어를 실행하지 않는다.
  - ident 프로토콜은 HTTP 트랜잭션을 유의미하게 지 연시 킨다. • 방화벽이 ident 트래픽이 들어오는 것을 막는 경우가 많다.
  - ident 프로토콜은 안전하지 않고 조작하기 쉽다.
  - ident 프로토콜은 가상 IP 주소를 잘 지원하지 않는다.
  - 클라이언트 사용자 이름의 노출로 인한 프라이버시 침해의 우려가 있다.
- 아파치 웹 서버: IdentityCheck 지시어를 통해 ident 룩업 사용 가능

## 5.5 단계 2: 요청 메시지 수신

- 커넥션 데이터 도착 시 웹 서버는 해당 데이터 읽고 파싱하여 요청 메시지 구성함

<img width="574" alt="image" src="https://github.com/user-attachments/assets/85f68bfe-71fc-4e42-877e-aa6193d49e2c" />

- 요청줄을 파싱하여 요청 메서드, 지정된 리소스의 식별자(URI), 버전 번호를 찾음
  - 각 값은 스페이스 한 개로 분리되어 있으며, 요청줄은 캐리지 리턴 줄바꿈 (CRLF) 문자열로 끝남
- 메시지 헤더들을 읽음, 메시지 헤더는 CRLF로 끝남
- 헤더의 끝을 의미하는 CRLF로 끝나는 빈 줄을 찾아냄 (존재한다면)
- 요청 본문이 있다면, 읽어 들임 (길이는 Content-Length 헤더로 정의된다)

- 네트워크 커넥션은 언제라도 무효화될 수 있음
  - 때문에 파싱해서 이해하는 것이 가능한 수준의 분량 확보 때까지 데이터를 네트워크로부터 읽어 메시지 일부분을 메모리에 임시저장할 필요 있음

### 5.5.1 메시지의 내부 표현

- 요청 메시지를 편리한 내부 데이터 구조로 파싱 (e.g. 룩업 테이블)

### 5.5.2 커넥션 입력/출력 처리 아키텍쳐

- 웹 서버 아키텍처 차이에 따라 요청 처리 방식 다름

**단일 스레드 웹 서버**

- 한번에 하나씩 요청 처리함
- 트랜잭션 완료 후 다음 커넥션 처리되며, 처리 도중 모든 다른 커넥션은 무시됨
- 성능 문제 야기하므로 로드가 적은 서버나 type-o-serve와 같은 진단 도구에서만 적당함

**멀티프로세스와 멀티스레드 웹 서버**

- 스레드/프로세스는 필요할 때 / 미리 만들어 사용함
- 몇몇 서버는 매 커넥션마나다 스레드/프로세스 하나 할당하기도 하는데 메모리나 시스템 리소스 많이 소비됨
  - 따라서 스레드/프로세스 최대 개수에 제한둠

**다중 I/O서버**

- 커넥션 상태 바뀌면 그 커넥션에 대해 작은 양의 처리 수행됨

**다중 멀티스레드 웹 서버**

- 멀티스레딩 + 다중화 결합한 방법
- 여러개의 스레드(보통 하나의 프로세스)는 각각 열려있는 커넥션 감시하고, 각 커넥션에 대해 조금씩 작업 수행함

## 5.7 단계 4: 리소스의 매핑과 접근

- 웹 서버 = 리소스 서버
  - HTML 페이지, JPEG 이미지 같은 미리 만들어진 콘텐츠 제공
  - 서버 위에서 동작하는 리소스 생성 애플리케이션 통해 만들어진 동적 콘텐츠 제공
- 콘텐츠 전달 전 요청 메시지의 URI에 맞는 콘텐츠 / 콘텐츠 생성기 웹 서버에서 찾아 콘텐츠의 원천을 식별해야함

### 5.7.1 Docroot

- 리소스 매핑에도 여러 종류 존재함
- 가장 단순한 형태는 요청 URI = 웹 서버의 파일 시스템 안에 있는 파일 이름으로 사용하는 것
- 문서 루트 = **docroot** = 웹 콘텐츠를 위해 예약한 웹 서버 파일 시스템의 특별한 폴더

<img width="596" alt="image" src="https://github.com/user-attachments/assets/f1bf7522-6d44-4fee-ac95-91113a9d08ae" />

- httpd.conf 설정 파일에 DocumentRoot 줄 추가해서 아파치 웹 서버의 문서 루트 설정할 수 있음
- 서버는 상대적인 url이 docroot를 벗어나 파일 시스템의 docroot 이외 노출되는 일 생기지 않게 해야함 (`http://docroot/../`)

**가상 호스팅된 docroot**

- 가상 호스팅 웹 서버: 사이트에 그들만의 분리된 문서 루트 줌, 한 웹 서버에서 여러 개의 웹 사이트 호스팅
- URI / Host 헤더에서 얻은 **IP 주소**나 **호스트 명**을 이용해 올바른 문서 루트 식별

- 서로 다른 사이트 주소 > HTTP Host 헤더나 서로 다른 IP 주소를 이용해 구분할 수 있음

- 아파치 웹 서버의 경우 각 가상 웹 사이트의 VirtualHost 블록이 가상 서버에 대한 DocumentRoot 지시자를 포함하도록 설정해야함

**사용자 홈 디렉터리 docroots**

- 사용자들이 한 대의 웹 서버에서 각자 개인 웹 사이트를 만들 수 있도록 해줄 수도 있음
- 빗금(`/`), 물결표(`~`) 다음 사용자 이름 = e.g. `/sujong`, `~sujong`로 시작 URI = 그 사용자의 개인 문서 루트 가리킴
- 개인 docroot = 사용자 홈 디렉터리 안에 있는 `public_html`로 불리는 디렉터리임 (설정에 따라 다를 수도)

### 5.7.2 디렉터리 목록

- 웹 서버가 파일이 아닌 디렉터리에 대한 URL 요청도 받을 수 있음
- 디렉터리 URL 요청 수신 시 대부분 취하는 행동

  - 에러 반환
  - 디렉터리 대신 특별한 `색인 파일` 반환
  - 디렉터리 탐색해서 그 내용을 담은 HTML 파일 반환

- 어떤 디렉터리 내 index.html, index.htm 존재할 시 해당 파일 콘텐츠 반환함

1. 사용자가 웹사이트의 폴더(디렉터리) URL을 요청
2. 서버는 자동으로 해당 폴더에서 특정 기본 파일을 찾음

- 우선순위: index.html → index.htm → home.html → home.htm → index.cgi

3. 해당 기본 파일을 발견하면 그 파일의 내용을 보여줌
4. 기본 파일이 없으면, 폴더 내 모든 파일 목록을 보여줄 수 있음

- 일반적으로 발견할 수 없는 파일도 들어날 수 있음

### 5.7.3 동적 콘텐츠 리소스 매핑

- URI ↔️ 동적 리소스에 매핑 가능
- 요청에 맞게 콘텐츠를 생성하는 프로그램에 URI 매핑하는 것

<img width="597" alt="image" src="https://github.com/user-attachments/assets/94c2ef40-39a1-4205-8bf4-4aafa0f66f07" />

- 아파치 웹 서버의 CGI(Common Gateway Interface) 설정에
  - URI 경로를 실행 가능한 프로그램 디렉터리로 매핑
    - 예: /cgi-bin/ URI는 /usr/local/etc/httpd/cgi-programs/ 디렉터리의 프로그램 실행
  - 특정 확장자 파일 실행 설정 가능
    - 예: .cgi 확장자 파일은 실행되도록 설정
  - CGI는 초기 서버사이드 애플리케이션 인터페이스
    - 현대에는 ASP, 자바 서블릿 등 더 강력한 기술로 대체됨

> Q. 갑자기 애플리케이션 서버, 웹 서버, 백엔드 애플리케이션의 차이점이 헷갈림

### 5.7.4 서버사이드 인클루드 (Server-Side Includes, SSI)

- **서버사이드 인클루드(SSI)**: 웹 서버가 특정 HTML 주석 패턴을 검사하고, 이를 변수 값이나 스크립트 출력으로 대체하는 기술
- 페이지에 동적 콘텐츠를 쉽게 삽입할 수 있음

### 5.7.5 접근 제어

- 각 리소스에 접근 제어 할당 가능
- 클라이언트의 IP 주소에 근거해 접근 제어 가능, 또는 비밀번호를 물어볼 수도 있음

## 5.8 단계 5: 응답 만들기

- 서버: 리소스 식별 > 요청 메서드로 서술되는 동작 수행 > 응답 메시지 반환
- 응답 메시지: 응답 상태 코드, 응답 헤더, 응답 본문 (생성 시에만) 포함

### 5.8.1 응답 엔터티

- 트랜잭션이 응답 본문 생성 시 응답 메시지가 포함하는 내용
  - 응답 본문의 MIME 타입을 서술하는 Content-Type 헤더
  - 응답 본문의 길이를 서술하는 Content-Length 헤더
  - 실제 응답 본문의 내용

### 5.8.2 MIME 타입 결정하기

**mime.types**

- 파일 확장자를 기반으로 MIME 타입을 결정하는 방법

**매직 타이핑(Magic typing)**

- 파일 내용을 검사해 MIME 타입을 찾는 방법
- 표준 확장자 없이 이름 지어진 경우 편리

**유형 명시(Explicit typing)**

- 특정 파일이나 디렉터리의 MIME 타입을 직접 설정할 수 있는 방법

**유형 협상(Type negotiation)**

- 웹 서버는 리소스의 최적 문서 형식과 MIME 타입을 사용자와 협상을 통해 결정 가능
- 한 리소스가 여러 종류의 문서 형식에 속하도록 설정 가능
- 특정 MIME 타입 설정도 가능

### 5.8.3 리다이렉션

**영구히 리소스가 옮겨진 경우**

- 리소스가 영구적으로 이동된 경우, 301 Moved Permanently 상태 코드를 사용함

**임시로 리소스가 옮겨진 경우**

- 리소스가 임시로 이동된 경우, 303 See Other 또는 307 Temporary Redirect를 사용함

**URL 증강**

- 상태 정보를 포함한 새 URL로 리다이렉트하여 트랜잭션 상태를 유지함

**부하 균형**

- 과부하된 서버는 클라이언트를 덜 부하된 서버로 리다이렉트할 수 있음

**친밀한 서버로 리다이렉션**

- 서버는 사용자 정보를 가진 다른 서버로 리다이렉트할 수 있음

**디렉터리 이름 정규화**

- 슬래시(/)가 빠진 디렉터리 요청을 수정하여 올바른 경로로 리다이렉트함

## 5.9 단계 6: 응답 보내기

- 서버는 여러 클라이언트에 대한 많은 커넥션 가질 수 있음
- 서버는 커넥션 상태 추적해야함, 지속 커넥션 특별히 주의, 비지속 커넥션은 서버가 모든 메시지 전송 후 커넥션 닫음

## 5.10 단계 7: 로깅

- 트랜잭션 완료 시 웹 서버는 트랜잭션 수행에 대한 로그를 로그 파일에 기록함
